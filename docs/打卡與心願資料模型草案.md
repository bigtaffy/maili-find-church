# 打卡與心願資料模型草案

## 目的

這份文件定義第一階段「無會員、本機版」的資料模型。

目標是讓前端在開始實作前，先把：

- 型別
- localStorage 結構
- 狀態值
- 基本驗證規則

整理成固定格式，避免後續反覆修改資料形狀。

## 第一階段前提

- 無會員
- 無後端同步
- localStorage 為正式保存位置

## 型別總覽

第一階段建議建立以下型別：

- `PilgrimageStamp`
- `PilgrimageStampMap`
- `WishCategory`
- `WishStatus`
- `PilgrimageWish`
- `PilgrimageWishSlots`
- `PilgrimageWishMap`
- `PilgrimageUiState`

## 1. 朝聖印章

### 型別

```ts
export type PilgrimageStamp = {
  parish_id: number;
  stamped_at: string;
  lat: number;
  lng: number;
  accuracy?: number | null;
  verification_method: 'gps' | 'manual';
};
```

### Map 型別

```ts
export type PilgrimageStampMap = Record<string, PilgrimageStamp>;
```

### 設計說明

- key 使用字串版 `parish_id`
- value 是該教堂唯一一筆印章
- 第一階段不記錄多次到訪歷史

### 驗證規則

- `parish_id` 必須存在
- `stamped_at` 必須是 ISO 字串
- `verification_method` 僅允許 `gps` 或 `manual`

## 2. 心願卡

### 解鎖規則

- 只有該教堂已存在朝聖印章時，才視為心願卡已解鎖
- 若該 `parish_id` 尚未蓋章，前端不應提供心願卡寫入入口
- 是否解鎖可直接由 `hasPilgrimageStamp(parishId)` 判斷，不需要額外存一個 `unlocked` 欄位

### WishCategory

```ts
export type WishCategory = 'self' | 'family' | 'world';
```

### WishStatus

```ts
export type WishStatus = 'pending' | 'fulfilled' | 'released';
```

### 單張心願卡

```ts
export type PilgrimageWish = {
  category: WishCategory;
  content: string;
  status: WishStatus;
  created_at: string;
  updated_at: string;
};
```

### 三格槽位

```ts
export type PilgrimageWishSlots = {
  1: PilgrimageWish | null;
  2: PilgrimageWish | null;
  3: PilgrimageWish | null;
};
```

### 教堂心願卡集合

```ts
export type PilgrimageWishMap = Record<string, PilgrimageWishSlots>;
```

### 設計說明

- 每間教堂固定 3 格
- slot 不做動態新增
- 類別和 slot 可先固定對應，也可讓使用者選

## 3. UI 狀態

### 型別

```ts
export type PilgrimageUiState = {
  stamp_intro_seen?: boolean;
  wish_intro_seen?: boolean;
  wish_local_only_notice_seen?: boolean;
};
```

### 用途

- 控制提示是否顯示過
- 避免重複 onboarding

## 操作規則

### 蓋章

- 若該 `parish_id` 尚未存在，新增一筆
- 若已存在，不新增第二筆
- 第一階段視為「已蓋過」即可

### 儲存心願卡

- 只能對已蓋章的 `parish_id` 寫入
- 指定 `parish_id` 與 `slot`
- 若該教堂不存在，先建立 3 格空槽
- 寫入指定 slot
- `updated_at` 必須更新

### 清除心願卡

- 清除時僅把該 slot 設為 `null`
- 不要直接刪除整個 parish entry
- 若 3 格都為 `null`，可考慮刪除整個 parish key

## 建議的 storage module API

建議建立：

- `src/lib/pilgrimageStorage.ts`

### 讀取

```ts
getPilgrimageStamps(): PilgrimageStampMap
getPilgrimageStamp(parishId: number): PilgrimageStamp | null
getPilgrimageWishes(): PilgrimageWishMap
getPilgrimageWishesByParish(parishId: number): PilgrimageWishSlots
getPilgrimageUiState(): PilgrimageUiState
```

### 寫入

```ts
savePilgrimageStamp(input: PilgrimageStamp): void
savePilgrimageWish(parishId: number, slot: 1 | 2 | 3, wish: PilgrimageWish): void
clearPilgrimageWish(parishId: number, slot: 1 | 2 | 3): void
setPilgrimageUiState(patch: Partial<PilgrimageUiState>): void
```

### 查詢

```ts
hasPilgrimageStamp(parishId: number): boolean
getPilgrimageStampCount(): number
```

## 錯誤處理

### localStorage 讀取失敗

- JSON parse 失敗時回傳安全預設值
- 同時可以考慮清掉壞掉的 key

### localStorage 寫入失敗

- `QuotaExceededError` 時應給使用者提示
- 不應讓整個頁面崩潰

## 未來升級方向

若後續加入會員，可以在 `v2` 擴充：

- `remote_id`
- `synced_at`
- `device_id`
- `updated_source`

但第一階段先不要把這些欄位塞進資料模型。

## 第一階段結論

第一版資料模型要小、清楚、穩定：

- 一間教堂一枚印章
- 一間教堂三張心願卡
- 額外只有少量 UI 狀態

先讓資料結構定住，之後 UI 和 storage module 才會寫得順。
