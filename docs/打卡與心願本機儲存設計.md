# 打卡與心願本機儲存設計

## 前提

目前產品**沒有會員機制**，因此打卡與心願功能的第一版應以「本機保存」為主，不設計雲端同步。

這代表：

- 朝聖印章保存在目前裝置
- 心願卡保存在目前裝置
- 換手機、清除瀏覽器資料、換瀏覽器後，資料會消失

第一版應先把體驗做順，不要先把同步架構做重。

## 設計目標

- 讓使用者在沒有會員的情況下，也能使用朝聖印章與心願卡
- 避免重整頁面後資料消失
- 把資料集中管理，避免 localStorage key 四散
- 為未來會員同步預留版本升級空間

## 設計原則

- localStorage 是第一版正式保存位置
- key 必須有命名空間與版本
- 結構要簡單、穩定、可讀
- 不做待同步 queue
- 不做後端衝突處理
- 先支援單裝置體驗

## 儲存範圍

第一版 localStorage 只需要存這三類：

1. 朝聖印章
2. 心願卡
3. UI 狀態

## Key 命名規則

建議統一使用 `maili:` 前綴。

### 命名格式

`maili:{domain}:{resource}:v{version}`

第一版建議：

- `maili:pilgrimage:stamps:v1`
- `maili:pilgrimage:wishes:v1`
- `maili:pilgrimage:ui-state:v1`

## 1. 朝聖印章

key:

- `maili:pilgrimage:stamps:v1`

用途：

- 記錄哪些教堂已蓋章
- 記錄蓋章日期與當下座標

資料格式：

```json
{
  "632": {
    "parish_id": 632,
    "stamped_at": "2026-03-27T15:03:00+08:00",
    "lat": 25.0518,
    "lng": 121.5574,
    "accuracy": 24,
    "verification_method": "gps"
  }
}
```

設計重點：

- 用 `parish_id` 當 key，避免重複
- 同一間教堂只保留一筆有效印章
- 後續若想加回訪次數，可再擴充欄位

可擴充欄位：

- `visit_count`
- `last_seen_at`
- `stamp_style`

## 2. 心願卡

key:

- `maili:pilgrimage:wishes:v1`

用途：

- 保存每間教堂的 3 張心願卡

資料格式：

```json
{
  "632": {
    "1": {
      "category": "self",
      "content": "希望自己能更平靜",
      "status": "pending",
      "created_at": "2026-03-27T15:10:00+08:00",
      "updated_at": "2026-03-27T15:10:00+08:00"
    },
    "2": {
      "category": "family",
      "content": "希望家人平安",
      "status": "pending",
      "created_at": "2026-03-27T15:11:00+08:00",
      "updated_at": "2026-03-27T15:11:00+08:00"
    },
    "3": null
  }
}
```

設計重點：

- 以 `parish_id -> slot(1~3)` 做索引
- 每間教堂固定只有 3 格
- 空格可用 `null` 表示
- 不需要額外分草稿與正式版本

### category 建議值

- `self`
- `family`
- `world`

### status 建議值

- `pending`
- `fulfilled`
- `released`

## 3. UI 狀態

key:

- `maili:pilgrimage:ui-state:v1`

用途：

- 記錄是否看過引導
- 記錄某些一次性提示是否已顯示

資料格式：

```json
{
  "stamp_intro_seen": true,
  "wish_intro_seen": true,
  "wish_local_only_notice_seen": true
}
```

設計重點：

- 只存非常輕量的介面狀態
- 不要把業務資料塞進這個 key

## 不建議存進 localStorage 的資料

- 會員憑證
- 敏感個資
- 大量圖片
- 大型歷史紀錄
- 未來若出現的完整同步快照

## 讀寫策略

### 初始化

頁面進入時：

1. 讀取 `stamps`
2. 讀取 `wishes`
3. 讀取 `ui-state`
4. 用本機資料直接渲染畫面

### 蓋章

1. 使用者在教堂範圍內按下蓋章
2. 寫入 `stamps`
3. 立即更新 UI
4. 播放蓋章動畫

### 心願卡

1. 使用者輸入內容
2. 儲存時直接寫入 `wishes`
3. 立即更新 UI

## 清理與升級

### 清理

第一版不主動清理印章與心願卡。

可提供手動入口：

- 清除我的朝聖印章
- 清除我的心願卡

### 版本升級

若未來要接會員同步或改資料結構：

- 升級 key 版本，例如 `v1 -> v2`
- 啟動時可做 migration
- 若 migration 太複雜，可保守清掉舊資料並提示使用者

## UI 提示建議

因為目前沒有會員，建議在第一次使用時清楚提示：

- 目前朝聖印章與心願卡會保存在此裝置
- 若更換裝置或清除瀏覽器資料，內容將不會保留

這段提示應在功能入口附近就看得到，不要藏在設定頁。

## 建議抽成單一 storage module

建議建立：

- `src/lib/pilgrimageStorage.ts`

集中處理：

- key 常數
- parse / stringify
- schema 驗證
- 讀寫方法

### 建議方法

- `getPilgrimageStamps()`
- `savePilgrimageStamp()`
- `hasPilgrimageStamp(parishId)`
- `getPilgrimageWishes(parishId)`
- `savePilgrimageWish(parishId, slot, wish)`
- `clearPilgrimageWish(parishId, slot)`
- `getPilgrimageUiState()`
- `setPilgrimageUiState()`

## MVP 結論

第一版 localStorage 的責任很單純：

1. 保存朝聖印章
2. 保存心願卡
3. 保存功能提示狀態

先把單機版體驗做順，等未來真的有會員機制，再往雲端同步升級。
